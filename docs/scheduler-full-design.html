kiro-rs 调度完整改造设计

这份设计不再追求“最小能跑”，而是按上线标准收敛成一版完整方案： 调度主脑参考 CPA，并发槽位参考 sub2api，实际落地面只在 kiro-rs，不改线上 sub2。

目标：完整可上线 范围：kiro-rs 内部改造 输入：生产日志 + CPA + sub2api 并发实现 日期：2026-05-19

文档定位

这份文档现在同时承担两件事：一是定义调度内核和接口，二是给出管理面的 UI 原型展示。后续只维护这一份。

设计目标

这里直接写死主需求，避免后面再被兼容字段、局部展示或测试细节带偏。
主目标 定稿口径
调度主脑 kiro-rs 调度参考 CPA 的粘性路由，按会话稳定命中同一账号；命中风险时自动脱粘。
并发窗口 每个账号增加 sub2api 风格的并发上限、槽位占用与释放路径，解决“账号能用但被打爆”的问题。
下游兼容 保留 sub2 需要的字段和展示口径，但不让 sub2 反向决定底层调度。
管理面 补单账号测试和运行态可视化，交互结构参考 sub2，视觉语言沿用现有 kiro-rs admin-ui。
版本治理 管理面可补版本检测、发布说明入口与部署方式提示；是否同时纳入自动更新能力，取决于部署方式与运维边界。

前提边界

下面这些边界先冻结，后续实现都按这个范围走，不再反复摇摆。
  • kiro 账号本体没有可直接复用的 5h / 7d 账号窗口语义，这里的“窗口限制”只解释成账号并发窗口。
  • 生产日志已经证明，真正的底层问题是 429 与 suspicious activity 后缺少摘流，而不是单纯“没号了”。
  • 线上 sub2 不改；参考 sub2 只为了并发槽位、测试交互、兼容参数和视觉口径。
  • UX 范围以本稿为准，后续实现阶段在这套结构内细化，不另起一套后台。
  • 单账号测试保留 sub2 的功能结构，但视觉必须贴近现有 kiro-rs admin-ui。
  • 不引入 kiro 上游不支持的能力，不把本地状态误写成上游真实能力。
  • 批量测活只作为巡检补充，不替代单账号测试。
  • sub2 已有版本检测与在线更新链路；本项目若引入同类能力，需要先区分“版本信息展示”和“自动替换重启”两类范围。
  • 源码运行、容器运行、二进制路径不可写、无重启托管时，在线更新能力是否可落地，取决于实例部署条件。

参考源码结论

来源 能借什么 不借什么
CPA 粘性路由主脑、会话命中优先、失败后脱粘、状态驱动选号 不直接照搬 provider/model shard 复杂度，但它就是本次调度主脑参考
sub2api 账号槽位、等待队列、占槽释放时机、单账号测试交互、管理面展示口径 不把它的选号逻辑、不把 OpenAI 特有语义硬套到 Kiro
sub2 更新链路 版本徽标、GitHub Release 检测、更新日志入口、Linux 单机场景下的重启/回滚思路 需要结合本项目部署方式评估是否能复用整套自更新流程
superman2003/sub2api Kiro 相关 usage 展示、prompt cache 视觉补偿 不把它当作调度主骨架

源仓库 PR 调研结论

截至 2026-05-19，已核对 hank9999/kiro.rs 的 open / closed PR。结论很明确：上游确实已经有人补过一批调度、管理端和验证能力，但这些改动大多是局部增强，还没有覆盖本次需求里的“CPA 式粘性路由 + sub2 式并发槽位 + Kiro 实际限频语义”这一整套闭环。
PR 结论 可借鉴点 不足点 本项目取舍
#63
负载均衡和批量管理		 已合并 把多账号调度、统计持久化、批量管理、管理面基础入口正式引入主线 当时的 balanced 还是“最少成功次数”思路，没有粘性路由、没有并发槽位、没有冷却回退 承认它是上游调度起点，但不够支撑当前需求
#124
balanced 改随机选择		 未合并 说明上游已有共识：success_count 轮询太僵，纯最少次数不够自然 只是把“最少次数”换成“随机”，依然没有回答同一会话该粘哪一个、并发满了怎么办、全池冷却怎么办 不直接采用；随机只能是兜底，不是主调度策略
#136
API key 后缀指定凭证		 未合并 允许外部系统显式命中某个账号，本质上接近“外部 pin” 需要下游主动传账号 ID；而当前线上对外只有一个 key，不能靠它完成 CPA 式粘性 可保留为调试/运维后门，但不能作为主方案
#140
RPM limit		 进行中 补了限流规则、运行态、管理端编辑，以及“全体限流时等待最早恢复”这一层概念 核心假设仍然是“预定义窗口限流”。这更像 OpenAI / sub2 的规则系统，不是 Kiro 线上真实语义；日志已证明 Kiro 可以超额继续打，也不存在固定 5h / 7d 窗口 只借展示字段和状态表达，不把它的窗口模型当底层调度真相
#111
刷新失败禁用策略		 已合并 把 refresh_failure_count 和普通调用失败拆开，禁用原因更清晰 它处理的是“Token 刷新连续失败”，不是“请求命中 429 / suspicious 后如何继续调度”；仍然偏硬禁用思路 字段和原因拆分值得保留，但不能替代冷却回退策略
#108
单账号强制刷新		 未合并 说明上游也在补“单账号可操作性” 属于运维动作，不解决调度主脑问题 可作为后续管理面补充，不进入本次调度主线判断
#40
凭据验证 / 单账号验证		 进行中 单账号验证入口、批量验证与结果分类，这和“测试一个账号到底能不能打”方向一致 验证本身不参与真实调度，不回答粘性命中、并发占槽、限频回退 交互可以借，语义仍按“单账号测试”处理，不包装成调度修复
#163
凭据管理增强		 进行中 编辑、重登录、代理池、批量导入细节，适合增强管理面 侧重点是账号管理，不是调度选路和运行态退化策略 可单独吸收管理能力，不把它误判成调度修复
#34
凭据卡片样式与余额刷新		 未合并 说明上游前端有人持续补视觉和卡片信息密度 只改善显示，不改变调度行为 视觉不照搬，继续按当前 kiro-rs 前端骨架做增量改造
上游 PR 里最接近本次需求的是 #63、#124、#136、#140、#40。但它们拼起来仍然缺三块关键能力：第一，会话级粘性路由；第二，账号并发槽位；第三，全池冷却时的软回退而不是直接宣告“全灭”。
这也和本轮真实验证结果一致：test 实例已经承接线上流量，但当 19 个账号同时进入 cooldown 后，当前实现会把这些账号全部摘出可选集合，最终出现 ready_ids = []，并报“所有凭据均已禁用（19/20）”。这不是上游已有 PR 能直接解决的问题。
  • 因此，本项目后续不应把“窗口限流配置”当成调度核心，而应把它降为展示层或附加约束层。
  • 真正的核心仍然是：先按 CPA 风格做粘性选号，再叠加 sub2 风格并发槽位，最后在 Kiro 返回 429 / suspicious 时走软冷却和风险回退。
  • 单账号测试、卡片状态翻译、运行态展示、批量恢复这些能力，上游 PR 可以提供局部参考，但不能替代调度主脑本身。

完整状态模型

完整方案依然只是几个字段，只是要把它们放进明确的状态层里，而不是散着写。
层 字段 说明
账号静态配置 priority、maxConcurrent、supportsOpus、disabled 来自配置文件或管理面板，重启后仍存在
账号运行态 inflight、cooldownUntil、last429Kind、recent429Count、recentSuspiciousCount、refreshFailureCount 只在进程内维护，决定调度器当前能不能选这个账号
会话粘性态 sessionKey -> accountId、boundAt、expiresAt 用于会话稳定路由；若上层传显式 pin，则先消费 pin
请求态 requestId、triedAccountIds、selectedAccountId 单次请求内的防重复撞击
展示兼容态 currentLoad、maxConcurrent、cooldownRemainingMs、stickyHit 给 sub2 参数兼容和前端展示，不作为底层真相

账号状态机

ready
  -> inflight 满 -> saturated
  -> 普通 429 -> cooldown(short)
  -> suspicious -> cooldown(long)
  -> refresh 429 连续失败 -> refresh_blocked / disabled
  -> 403 suspended -> disabled

saturated
  -> inflight 释放 -> ready

cooldown(short/long)
  -> 到时 -> ready
  -> 冷却中再命中更重错误 -> 延长冷却 / 升级

refresh_blocked
  -> 达阈值 -> disabled
  -> 人工恢复或后续成功刷新 -> ready

disabled
  -> 仅人工启用或显式恢复

选号顺序

选择顺序必须固定，不允许“报错了再临场猜”。 
Layer 1: 硬过滤
- disabled
- cooldownUntil > now
- inflight >= maxConcurrent
- 模型不兼容
- 本次请求已经试过

Layer 2: 显式 pin 命中
- 上层已指定 preferred / pinned account
- 该账号仍满足 Layer 1
- 命中则直接选

Layer 3: 粘性命中
- 有 sessionKey
- sticky 账号仍满足 Layer 1
- 命中则直接选

Layer 4: 全局选号
- 先按 priority 分组
- 同优先级内按 round-robin
- 再过一层并发槽位判断
- 若启用负载感知，则优先 inflight 更低的账号

Layer 5: 失败回退
- 当前账号写入 triedAccountIds
- 重新从 Layer 1 开始
- 单次请求不重复打已试账号

粘性路由规则

规则 处理
上层显式 pin 优先于 session 粘性；调度器先判断 pin 账号当前能否接单
新会话第一次请求 正常选号，成功后绑定 sessionKey -> accountId
命中粘性账号且账号健康 直接复用，保持会话稳定
粘性账号并发满 脱粘本次请求，走全局选号；可保留映射短时间不删
粘性账号命中普通 429 当前请求脱粘，账号进入短冷却
粘性账号命中 suspicious 立即脱粘并解绑，账号进入长冷却
粘性账号被禁用 / suspended 立即解绑，后续不再尝试命中
粘性 TTL 带过期时间，避免陈旧绑定长期残留

并发槽位设计

这部分明确借 sub2 的实现思路，但只借“并发门”，不借它的选号主脑。

占槽时机

  • 不是“选中就占”
  • 而是准备真正发请求时占

释放时机

  • 成功释放
  • 失败释放
  • 超时释放
  • 流式中断也释放

等待队列

  • 可按阶段决定是否提供阻塞等待
  • 完整方案包含账号级等待计划
项 需求描述
maxConcurrent 每账号独立配置，默认值可以统一设定
inflight 内存原子计数或锁保护计数
等待计划 可返回“等待时长 / 当前排队人数”给前端或兼容层
单请求保护 一个请求绝不重复占同一账号第二次

错误分类与状态转移

错误 运行态处理 是否脱粘 是否禁用
普通 429 短冷却，累积计数 是 否
suspicious activity 长冷却，必要时提高后续冷却时长 是 否，先不直接禁用
refresh 429 走刷新失败计数，与 API 429 分开 是 达到阈值后是
403 suspended 直接标记不可调度 是 是
400 Improperly formed request 只记请求错误，不污染账号健康 否 否
网络错误 / 5xx 短暂回退，不直接污染账号 按策略可不脱粘 否

sub2 参数兼容层

兼容层的职责是“把 kiro-rs 的真实运行态翻译成 sub2 看得懂的字段”，不是反过来决定底层调度。sub2 在这里是消费者，不是调度器。
兼容字段 来源 用途
currentLoad inflight 前端显示当前账号忙不忙
maxLoad maxConcurrent 显示并发上限
cooldownRemainingMs cooldownUntil - now 显示多久后可以再试
lastRateLimitKind last429Kind 前端区分普通限频和风控限频
stickyHit pin 或 session 粘性命中结果 辅助调试和观测
dispatchState 账号状态机 前端用颜色或标签表达当前状态

前端可视化

账号列表

  • 显示状态：可用 / 冷却中 / 并发满 / 禁用
  • 显示并发：inflight / maxConcurrent
  • 显示最近限频类型

详情侧栏

  • 最近 429 次数
  • 最近 suspicious 次数
  • 冷却剩余时间
  • 当前粘性会话数

用户感知

  • 只讲“当前忙、稍后再试、已切换其他账号”
  • 不暴露内部实现词汇

版本信息

  • 显示当前版本、最新版本、发布时间和更新说明入口
  • 根据部署方式展示“可更新”或“仅提示升级步骤”

版本检测与更新能力

调研结果显示，sub2 已经把版本检测、在线替换、重启、回滚拆成了完整链路。这里先按能力项把需求列清楚，不替实现阶段先做取舍。
当前实现口径：版本信息入口收进右上角轻量按钮，点击后进入弹窗查看当前版本、最新版本、部署方式、升级提示和发布说明链接，不再占用页面底部悬浮区。
能力 需求描述 依赖条件
版本检测 管理面显示当前版本、GitHub 最新 Release、发布时间、更新说明链接、是否落后 需要服务端拿到当前构建版本，并能访问 GitHub Release 元信息
升级提示 按部署方式展示升级路径，例如源码部署、二进制部署、容器部署分别展示不同说明 需要实例能识别部署模式，或允许管理员手动配置部署模式
在线更新 管理面触发下载 release、备份当前二进制、替换文件并发起重启 需要可写二进制路径、明确的重启托管方式，以及实例对自替换当前进程的运行权限
回滚 管理面可选择最近备份版本并触发回滚 需要完整的备份保留、版本元信息记录与重启能力
调研边界：如果实例是源码运行、容器运行、二进制路径不可写，或重启由外部系统托管，那么“在线更新”只能作为条件式能力存在，不能按统一可用能力描述。

单账号测试

批量测活只能做巡检，不能回答“这个账号此刻到底能不能打”。真正有用的是单账号测试，而且入口必须直接放在账号卡片上。
项 设计口径
主入口 每张账号卡片直接提供 测试接入 按钮，不藏到二级页面
测试目标 验证指定账号本身是否还能发起真实请求，并持续观察流式输出；这里必须固定命中该账号，不能自动切到别的账号
输出结果 像控制台一样实时展示开始、连接、输出、完成或报错，而不是等结束后一次性回填
交互要素 必须有模型选择、开始测试、重试、关闭；图像模型场景还要能填写提示词并预留结果预览区
边界 不把它包装成“上游恢复”或“长期稳定性证明”；它只证明当前这一次接入结果，同时允许绕过本地冷却/并发门去直接探测该账号
批量巡检 仍可保留，但只作为运维辅助，不替代单账号判断

前端展示口径

枚举值不要直接把 ready、cooldown、suspicious_activity 这种内部值扔给用户。页面默认显示中文含义，必要时再用 hover 提示补充说明。
当前实现已经按这个口径落地：状态、错误类型、禁用原因都优先显示中文；英文内部值不直接暴露，说明只放在 hover 或说明文案里。
状态 展示文案 展示方式
ready 就绪可用 绿色 Badge，展示 currentConcurrent / maxConcurrent
saturated 并发已满，等待释放 黄色 Badge，不额外起子卡片
cooldown + 普通 429 冷却中（普通限频） 蓝色文字或 outline Badge，显示倒计时
cooldown + suspicious 冷却中（风控限频） 红色风险 Badge，补充“已解除会话绑定”
blocked 刷新失败阻塞 黄色 Badge，显示 refreshFailureCount 和恢复按钮
disabled 已禁用 沿用当前 destructive / outline 组合

接口口径

这里必须跟项目现状对齐。前端 axios 基础路径已经是 /api/admin，现有风格也是对账号动作使用 POST。
接口 用途
GET /api/admin/credentials 沿用现有接口，只是把扩展运行态字段一起返回
POST /api/admin/credentials/:id/max-concurrent 调整账号并发上限
POST /api/admin/credentials/:id/test 对单个账号发起 SSE 流式测试，请求体可带 model_id、prompt，前端持续消费事件

POST /api/admin/credentials/:id/disabled 沿用现有启用 / 禁用开关
GET /api/admin/system/version 返回当前版本、最新 release、发布时间、更新日志链接、部署模式、是否具备在线更新条件
POST /api/admin/system/version/check 手动触发检查最新版本；只涉及 release 元信息，不涉及账号缓存

设计说明

  • 状态优先级：disabled > blocked > cooldown > saturated > ready
  • 颜色语义：绿色表示健康，黄色表示警告，蓝色表示冷却信息，灰色表示禁用，红色表示高风险
  • 并发上限支持在线调整并即时生效
  • 冷却倒计时由前端递减，到 0 后自动刷新状态
  • 普通 429 和 suspicious 必须在展示上明确区分，不能只写“限频”
  • blocked 状态的“手动恢复”只针对刷新失败阻塞，不针对 suspended
  • 版本入口放在顶部轻量区域，避免挤占账号调度主视图
  • 在线更新相关按钮是否可用，取决于实例是否满足更新条件

验收标准

目标 验收口径
429 摘流有效 同账号 suspicious 再次命中间隔不再是亚秒级集中
并发槽位有效 账号不会超过配置并发上限
粘性可控 健康时能稳定复用，出错时能自动脱粘
请求内不重复撞击 一次请求重试不会二次命中同一账号
展示可解释 前端能看到当前状态、并发、冷却，而不是只看到“失败”
版本信息可核对 管理面能展示当前版本、最新版本和更新条件，且与实际部署方式一致
